It's Never Just Three Clicks

Why a non-database connector (Salesforce, Workday) is a fundamentally different engineering problem than the database connector you just watched Niraj set up — and why that difference is what makes the combined pipeline worth selling as one story.
30 MIN SESSION PRESENTER: JASON CHLETSOS LIVE IN FIVETRAN UI CLICK A CARD FOR FULL-SCREEN DETAIL

00The premise

A database exposes a schema. An application exposes an opinion about how its product works. Those aren't the same problem, and the Fivetran UI is simple on purpose in both cases — which is exactly why it's easy to assume the underlying difficulty is the same too. It isn't.

DATABASE CONNECTOR

  • Fixed, introspectable schema (information_schema)
  • Auth = host / port / user / password
  • Change data via WAL / binlog / CDC log
  • DELETE means gone
  • One customer's Postgres looks like every other Postgres
vs.

SAAS / API CONNECTOR

  • Schema is a moving object graph, partly custom per org
  • Auth = OAuth, connected apps, permission sets, IP allowlists
  • Change data via timestamp-chasing or full snapshots
  • "Deleted" often means soft-deleted, flagged, or recycled
  • Every customer's org is configured differently

02Salesforce — seven things happening under the green checkmark

Walked through live, in setup-flow order. Click a card for the detail and mini-illustration to cite out loud.

OAuth against a connected app, a dedicated integration user, a specific permission set, IP allowlisting, API access enabled per-profile. Most of this is configured (or missing) before Fivetran ever appears.

Illustrative — profile config an admin has to get right first
ProfileIntegration API User
API Enabled (profile)off by default — must be granted
Login IP Rangesrestricted to office CIDR
Connected App Scopesapi, refresh_token
The credential is a pointer into an access model — not the access model itself. A DB connector needs none of this scaffolding.

Standard objects, custom objects, custom fields, lookup and master-detail relationships, junction objects, record types that make one physical table behave like several logical entities.

Illustrative — one org's slice of the object graph
Account→ Contact→ Opportunity→ OpportunityLineItem
Account→ Custom_Object__c→ Junction__c
Two chains shown. A real org runs hundreds of standard objects plus however many custom objects this customer built.
A CREATE TABLE is a contract. A Salesforce org is a living configuration.

Computed server-side, not stored like native fields. Roll-up recalculation has historically not reliably bumped the parent's SystemModstamp — a pure timestamp-based sync can miss the change.

Illustrative — a roll-up change that incremental sync can miss
OpportunityLineItem.Quantitychanged 10:02:03
Opportunity.Amount (roll-up)recalculated
Opportunity.SystemModstampstill 09:47:11 — unchanged
Sync polling on SystemModstamp windows never re-selects this Opportunity.
The bug that doesn't show up in a demo — it shows up eight months into production.

Recycle bin semantics via IsDeleted, silently excluded from standard queries. Capturing deletes correctly needs queryAll-style semantics or dedicated endpoints, and it varies by object.

Illustrative — same object, two query semantics
SELECT Id, Name FROM Account → 2 rows returned SELECT Id, Name FROM Account ALL ROWS → 3 rows returned (includes "Acme Corp (Old)", IsDeleted = true)
Salesforce has a parallel universe of "gone but not gone" — and you have to know to ask for it.

Extraction via SystemModstamp windows or object-specific change endpoints, with Bulk API v2 vs. REST chosen per object based on volume — and re-evaluated as the object grows.

Illustrative — capture latency, WAL vs. polling
Database (WAL) — change written 10:02:00
captured 10:02:00 — 0 min lag
Salesforce (poll every 15 min) — change written 10:02:00
waiting for next poll window
10:15
Same change, 13 minutes of unavoidable lag baked into the extraction model — not a bug, a property of polling an API.
Databases replicate. Salesforce has to be asked, over and over, forever.

Daily API call allotment tied to license edition and user count, shared with every other integration the customer runs. Careless batching doesn't just slow Fivetran — it can rate-limit the customer's other vendors.

Illustrative — one org's daily API budget, shared

A shared-commons problem, not a performance-tuning problem — burn the budget and someone else's integration gets throttled too.

Multi-select picklists (delimited strings), multi-currency amounts (value + currency code + historical conversion rate), compound fields, admin-renamed picklists that don't touch the API name.

Illustrative — raw API value vs. warehouse-ready shape
Raw: Region__c (multi-select)"East;West;Central"
Mapped3 junction rows (opportunity_id, region)
Raw: Amount + CurrencyIsoCode125000, EUR, rate 1.083 @ close date
Mappedamount_local, currency_code, amount_usd
A semantic modeling problem, not a type-casting problem — solved before dbt Wizard ever runs.
Land it: every one of these seven happens before dbt sees a row. That's Fivetran absorbing complexity so a customer's dbt models start from something coherent instead of raw API sludge.

03Workday RaaS — a different paradigm, not a harder version of the same one

Most integrations run through Reports-as-a-Service: an admin builds a custom report in Workday Studio, defines its fields and filters, and publishes it as a web service. That report is the schema — and it didn't exist until a human made it.

Illustrative — a report definition assembled by hand, before any connector exists
Worker_ID+ Legal_Name+ Position+ Cost_Center+ Effective_Date
Built in Workday Studio, then published as a web service. This assembly step is the schema. Nothing like it exists on the Salesforce or database side.
With Salesforce, Fivetran discovers a schema. With Workday, someone authors one first.

Integration System Users (ISU), tenant URLs that differ by environment (implementation/sandbox vs. production), credentials scoped per integration rather than per user.

Illustrative — same customer, two hosts
Tenant (sandbox)acme_impl-services1.workday.com
Tenant (production)acme_services1.workday.com — different host entirely
AuthSOAP + WS-Security, Integration System User
Credential scopeper-integration, not per-user
Not a login. A per-tenant integration contract that gets re-negotiated for every environment.

Most RaaS reports pull the full result set every time. Change detection often has to happen downstream — which is exactly the work Salesforce's SystemModstamp at least attempts natively.

Illustrative — data moved per sync, same-sized org
Salesforce incremental pull
unchanged, skipped
changed only
Workday RaaS pull
entire report, every run
The reconciliation work Salesforce does natively becomes someone's dbt model here.
Same-sized org, very different amount of data moved per sync — and very different downstream reconciliation cost.

Field-level masking and data residency are part of the connector conversation immediately, not an afterthought bolted on later.

Illustrative — same field, two viewers
Base_Pay — HR admin view$128,500
Base_Pay — integration/API viewmasked, role-dependent
Tenant data residencypinned to region
Every customer's Workday connector is bespoke — a contract against a report someone built, which can break silently if that report definition changes.
If Salesforce is "the API has opinions," Workday is "there is no API until a human builds one."

04Why this matters for NewCo

Not to make Fivetran look hard to use — the UI stays simple on purpose, and that's good engineering. The point is to stop treating Connections as the solved, boring part of the pipeline and Transformations as "the real work." The pitch is one pipeline: Connections absorbs this mess, Transformations (dbt Wizard) works well because the mess was already absorbed, Activations sends the result back out. Undersell ingestion and you set up your own transformation story to look worse than it is — when a customer asks why their dbt model output looks off, the honest answer is often further upstream than dbt.

The transferable checklist (works for NetSuite, Marketo, or whatever gets asked about live): auth model → object graph → deletes → incremental strategy → rate limits → type mapping. Every SaaS connector answers these six questions differently.
