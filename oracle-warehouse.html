Why an Oracle source and a Snowflake or Databricks destination hide as much configuration pain as any SaaS connector — the DBA tickets, network reviews, and CDC plumbing nobody sees before the first green checkmark.

00The premise

The database demo looks like the easy one — host, port, credentials, green checkmark, done. That's true on screen. It's true in the Fivetran UI because Fivetran already absorbed everything on the other side of this contrast before that screen ever loaded.

02Oracle as source — seven things a DBA has to sign off on first

Walked through in setup order. None of this shows up in the Fivetran connector form — it's the work that has to be finished before that form is worth filling out.

Getting from "here's the host and port" to a byte actually reaching Fivetran usually means a firewall ACL exception opened by a network team, a bastion host or SSH tunnel if the database has no public IP, and an Oracle Wallet / TLS certificate exchange if the listener requires encrypted traffic. None of that is a field in the Fivetran UI — it's a week of tickets across three teams before setup even starts.

Illustrative — what has to clear before a connection string means anything
Firewall exceptionticket to NetSec, 3–5 business days
Bastion / SSH tunnelrequired — DB has no public IP
TLS / Oracle Walletcustomer mandates encrypted listener
VPN / PrivateLinknegotiated per environment
The host/port fields are the last five minutes of a process that starts in someone else's ticket queue.

Log-based CDC needs the integration user to hold privileges most DBAs are cautious about handing out: SELECT on every table in scope (or SELECT ANY TABLE), EXECUTE on DBMS_LOGMNR, the LOGMINING role (12c+), and sometimes SELECT ANY TRANSACTION or FLASHBACK ANY TABLE for reconciliation. Every one of those triggers a "why does an ETL tool need that" conversation with security.

Illustrative — a grant list working its way through approval
SELECT ANY TABLErequested, pending DBA sign-off
EXECUTE DBMS_LOGMNRrequires LOGMINING role (12c+)
FLASHBACK ANY TABLEflagged by security review
Typical approval time1–3 weeks, not 1–3 clicks
This is the actual bottleneck on go-live — and it happens entirely outside Fivetran.

Log-based CDC depends on ARCHIVELOG mode being enabled, supplemental logging turned on at the database level (and often per-table for full row images), and enough archive log retention to survive a connector outage without a full resync. RAC adds shared-storage log merging across multiple redo threads.

Illustrative — prerequisites that predate the connector by years
ARCHIVELOG modemust be enabled — some prod DBs still aren't
Supplemental loggingDB + per-table — missing it means no CDC
Archive log retentionmust outlive the longest expected outage
RACN redo threads, log reads must merge across them
This isn't Fivetran's decision to make. It's downstream of a retention policy the customer set years before this call.

Oracle's multitenant architecture (Container Database + Pluggable Databases) means "the database" a customer names isn't always the database Fivetran actually connects to. Add synonyms pointing at objects in an entirely different schema, partitioned tables that look like one logical table but are N physical segments, and materialized views that are derived data masquerading as a source table.

Illustrative — what "the ORDERS table" can actually resolve to
CDB→ PDB: SALES_PROD→ Synonym: ORDERS→ Actual table: APPUSER.ORD_2024_P12
Four hops between the name in the connector config and the physical segment being read.
A schema browser has to resolve all of this correctly before "pick your tables" means anything.

A NUMBER column with no declared precision or scale can hold anything from an integer to a 38-digit decimal, so the connector has to sample or default rather than trust the column definition. RAW and LONG RAW are binary blobs with no defined encoding. TIMESTAMP WITH TIME ZONE and WITH LOCAL TIME ZONE carry offset semantics a naive cast throws away, and NCHAR/NVARCHAR2 columns can sit in a different character set than the rest of the database.

Illustrative — source type vs. what actually has to be decided
NUMBER (no precision/scale)sampled, not declared — could be int or decimal(38)
RAW(2000)binary, no encoding contract
TIMESTAMP WITH TIME ZONEoffset must survive the cast or history breaks
NVARCHAR2separate character set from the rest of the DB
None of this is a type-casting problem. It's a set of judgment calls the connector makes for the customer.

The historical sync of a large table is a full table scan against a system that's also serving live transactions. DBAs often insist it run in a maintenance window, or against a standby via Active Data Guard, instead of primary. LogMiner mining itself consumes CPU and PGA memory on the source instance — it isn't free just because it's "just reading logs."

Illustrative — why "just run the initial sync" isn't a one-line ask
Historical table scancompetes with live OLTP I/O
DBA requestmaintenance window, or read from standby
LogMiner miningconsumes CPU / PGA on the source instance
"Kick off the initial sync" is a scheduling negotiation with production, not a button.

Partition splits and merges, ALTER TABLE ADD COLUMN, and direct-path (APPEND) inserts can bypass or confuse log-based capture depending on version and configuration. Catching the drift often means re-snapshotting the table rather than quietly catching up — and that's another production-impact conversation, not a background task.

Illustrative — a DDL change during an active sync
ALTER TABLE ord_2024_p12 SPLIT PARTITION ... → partition boundary changes mid-sync → connector re-evaluates — may require a re-snapshot
The setup screen doesn't show what happens six months later when someone runs a partition maintenance job.
Land it: none of these seven show up in the five-field connector setup screen. They're the reason a "simple database connector" project plan has a DBA ticket queue as its critical path, not a Fivetran configuration step.

03Landing in Snowflake or Databricks — the setup on the other end of the pipe

The source isn't the only side with prerequisites. Getting data to actually land, safely and affordably, has its own six-item checklist — slightly different words depending on which warehouse the customer picked, same shape of problem.

Snowflake needs key-pair or OAuth authentication plus an IP allowlist / network policy, and often PrivateLink for a customer who won't allow traffic over the public internet. Databricks needs a service principal with Unity Catalog grants, workspace network configuration (VNet injection or PrivateLink), and cluster policies that restrict what a job is even allowed to spin up. Same shape of problem, different vendor console to file the ticket in.

Illustrative — provisioning before the first load job runs
Snowflakekey-pair auth + network policy allow-list
Databricksservice principal + Unity Catalog grant + workspace network rule
Either wayPrivateLink/VNet request, its own approval cycle
The destination side has its own security review running in parallel with the source side's.

In Snowflake, files land in an internal or external stage (S3, Azure Blob, GCS), then COPY INTO or Snowpipe loads them, with file format and compression settings that have to match on both sides. In Databricks, files land in cloud storage and Auto Loader or COPY INTO ingests into Delta tables, with schema inference or explicit hints deciding how Oracle's types actually land.

Illustrative — the hop between "extracted" and "queryable"
Oracle extract→ cloud storage (stage)→ COPY INTO / Auto Loader→ queryable table
Two extra hops between "the data left Oracle" and "an analyst can query it" — each with its own failure mode.

In Snowflake, warehouse size and auto-suspend/resume tuning matter: an XS warehouse makes a historical sync crawl, an oversized one burns credits sitting idle afterward. In Databricks, cluster autoscaling policy and instance type matter the same way — the wrong choice means paying for a large cluster mostly idle while it waits on network I/O from Oracle instead of doing compute work.

Illustrative — same historical backfill, two sizing mistakes
Snowflake warehouse too smallbackfill runs 8x slower, delays go-live
Snowflake warehouse too largecredits burned sitting idle after backfill
Databricks cluster oversizedpays for compute while waiting on Oracle I/O
Nobody put "estimate the backfill cost curve" on the project plan, but somebody's finance team notices when it's wrong.

Snowflake offers schema evolution on ingestion (auto-adding columns) plus VARIANT as an escape hatch for anything that doesn't map cleanly. Databricks has to choose between Delta schema enforcement and schema evolution mode on writes — get that setting wrong and a source DDL change either breaks the pipeline outright or silently drops data instead.

Illustrative — the same source change, two possible outcomes
Oracle: ALTER TABLE ADD COLUMNnew column appears mid-sync
Schema evolution ONcolumn added, backfilled as null
Schema enforcement ON, evolution OFFwrite rejected — pipeline halts
Whoever configures this setting is making a decision on behalf of every future Oracle DDL change, sight unseen.

Snowflake Streams and Tasks (or the merge logic the connector runs) apply CDC events in order and handle out-of-order delivery and deletes. Databricks leans on Delta's Change Data Feed and MERGE INTO to do the same job — but concurrent writers can hit optimistic-concurrency conflicts on the same table that need retry logic to resolve correctly.

Illustrative — three CDC events, one correct final state
10:02:01 INSERT id=501 10:02:04 UPDATE id=501 10:02:02 DELETE id=501 (arrives late, out of order) → naive apply: row 501 ends up present when it should be gone
Ordering bugs here don't throw an error — they just leave the warehouse quietly wrong.

Snowflake needs a role hierarchy plus tag-based masking and row access policies. Databricks needs Unity Catalog catalogs and schemas with table ACLs and column masking. If a customer's compliance team hasn't approved the access model before the pipeline goes live, the "successful sync" is itself the incident — sensitive Oracle data sitting in a warehouse with the wrong people able to see it.

Illustrative — what has to be true before go-live, not after
Role hierarchy / Unity Catalog grantsmust exist before first sync, not after
Column masking on PII fieldspolicy has to be attached pre-launch
Compliance sign-offmissing this turns a sync into an incident
Same story as the source side: the actual critical path is a decision made by people who never open the Fivetran UI.
The destination side doesn't get a pass just because "warehouses are easy." It's a parallel set of tickets running the whole time the source side's tickets are also running.

04Why this still matters for NewCo

Not to make the database story look impossible to sell — these pipelines get stood up successfully every week, and the demo really is that simple to click through. The point is the same one from the SaaS side, aimed at the opposite assumption: a "simple" Oracle-to-warehouse connector compresses months of DBA approvals, network reviews, and warehouse governance decisions into a five-field setup screen. If a prospect's team hears "three clicks" and assumes that's the whole project timeline, the miss that follows — a slipped go-live date — has nothing to do with Fivetran and everything to do with a security review nobody scoped up front.

The transferable checklist (works for SQL Server, Postgres, or MySQL, not just Oracle): network path → privilege grants → CDC mechanism → schema shape / datatype mapping → production impact → destination compute sizing → governance sign-off. Every one of these is a conversation with someone who isn't in the room.
